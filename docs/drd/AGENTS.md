# AGENTS.md - docs/drd

## DRD 是什么

DRD 是一份**业务决策记录**，不是数据契约。它的中心是**业务概念本身**，不是字段表。字段只是概念的一个侧面。

## 元认知

### 1. 字段是代码的事，业务决策是文档的事

代码已经告诉读者字段名和类型。DRD 不应该重复代码在做的事，而应该回答代码回答不了的问题：**这个字段为什么长这样**。

### 2. 读者是未来的人

DRD 最值钱的读者不是正在实现这个模块的开发者，而是半年后新来的团队成员。他看不懂的是你当初做决策的上下文，不是字段名。

### 3. 不记什么是决定文档生命力的关键

关系图、变更记录、数据约定——这些东西写的时候有用，但三个月后没人更新就变成错误文档。文档的权威性来自它的稳定性。

### 4. 责任边界比数据关系重要

一个字段为什么不在这个模型里，比这个字段是什么类型更能体现业务理解。职责边界应该明确：哪些事归这个模型，哪些事归其他模型。

### 5. DRD 的最佳形态是 onboarding 材料

用"怎么让新人最短时间理解这个业务概念"的标准来写 DRD，而不是用"怎么把数据结构描述清楚"的标准。
