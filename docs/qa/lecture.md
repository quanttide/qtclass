# 课时页面原型评审

## 产品视角

### 目标用户不明确 ✅ 已解决

描述和提纲使用文章式语言，未说明课程面向人群。没有区分「零基础开发者」还是「有经验工程师」。用户无法判断课程是否适合自己。

已补充目标用户说明：有 1 年以上编码经验的开发者，使用过 AI 辅助编程工具，开始感到代码质量不可控。

### 缺少学习目标与产出 ✅ 已解决

提纲列出 7 个要点，但未说明课程结束后用户能带走什么。缺少可衡量的学习目标。

已补充 3 条学习目标：识别代码腐化信号、掌握重构嵌入编码循环的工作方式、完成实战推演。

### 标题与描述脱节 ✅ 已解决

标题和描述围绕核心论点展开，提纲前 3 点都在铺垫问题，第 4 点才进入实战。铺垫过长，用户扫读时可能误判为观点文章而非课程。

已调整提纲，实战推演与技术债分析移至前 2 位，实例驱动认知。

### 缺少课时元信息 ✅ 已解决

没有课时时长、难度级别、前置要求。这些信息是用户选课决策的关键依据，当前完全缺失。

已补充课时信息到底部：时长 20 分钟、难度中级、形式视频+可运行示例。

### 实例不够具体 ⚠️ 保留

「用 Vibe Coding 写量潮课堂首页」作为实例方向合理，但未说明用户能看到的产出形式。缺少钩子——重构前后的对比效果或完整的组件示例，无法激发用户预期。当前未调整。

## 设计视角

### 视觉层级扁平 ✅ 已解决

标题、描述、目标用户、学习目标、提纲、课时信息——所有 section 的视觉权重几乎一致（h2 统一 1.1rem，卡片内间距相同）。用户扫读时找不到视觉锚点。

已处理：标题区独立为 header（无 section 边框），h2 改为小号大写 + 品牌色，描述改为引号卡片样式，形成三层视觉阶梯。

### 信息分组缺少视觉节奏 ✅ 已解决

6 个 section 从上到下依次堆叠，中间用分隔线等距隔开，阅读节奏单一。

已处理：分为 3 个逻辑区——标题区（header，无边框）、内容区（3 个 section，border-top 分割）、元信息区（底部 tag 栏，品牌色浅底）。

### 课时信息视觉权重过轻 ✅ 已解决

课时信息放在底部独立 section，本身是辅助决策的关键信息，但当前样式（灰色小字、flex 排布）视觉权重最低。

已处理：改用品牌色 pill 标签 + 浅蓝底通栏，形成底部视觉落点。

### 字体系统未定义 ✅ 已解决

全文只有一种字号体系，缺少标题、正文、辅助信息的分层定义。描述段落未做行长控制（max-width 未应用于段落），宽屏下阅读体验差。

已处理：建立字号阶梯（h1 1.625rem → 正文 1rem → 引号 0.9375rem → 标签 0.8125rem → section 标题 0.75rem），段落添加 max-width: 36em 控制行长。

### 缺乏品牌属性 ⚠️ 部分解决

当前是通用卡片样式，没有量潮课堂的品牌色、品牌字体、图标系统。

已补充 CSS 变量体系（brand / text / bg 色板），品牌色 #2563eb 贯穿页面。完整品牌系统需量潮课堂设计规范落地后补充（字体、图标、Logo、间距 scale）。

## 技术视角（Flutter + FastAPI）

### 数据模型与 API 未定义

当前内容写死在 HTML 中，未定义课时页的数据模型和 API 接口。Flutter 端需要课程 ID 来请求数据，FastAPI 端需要对应的 endpoint。

建议在 `docs/drd/` 中定义 Lecture 数据模型：id、title、description、target_audience、learning_objectives（数组）、outline（数组）、duration、difficulty、format。

### 课时内容以什么形式交付

「视频 + 可运行示例」中，视频是 URL 还是文件上传？可运行示例是代码片段（展示）还是完整的项目（可下载）？Flutter 端对两种形式的交互路径完全不同。

建议明确示例的交付形态，影响 Flutter 页面布局（示例是内嵌代码块还是跳转链接）。

### 页面状态覆盖

Flutter 页面通常需要处理加载态、空态、错误态。当前原型只展示了理想态的数据填充，未定义这三个状态下的界面表现。

建议为每个 section 设计加载中和加载失败的占位样式。

### 列表项交互缺失

提纲的 7 条在 Flutter 中大概率是可点击的（点击跳转到对应章节或展开详情），当前原型未体现任何交互状态——点击态、播放进度、完成状态。

建议为列表项补充交互状态定义。

## 业务视角（教学团队）

### 讲师信息缺失

当前页面没有展示授课讲师。教学团队需要讲师的品牌曝光来建立信任——姓名、头像、简介、相关课程列表。这既是教学团队的需求，也是学生的选课决策依据。

建议在标题区或底部增加讲师信息模块。

### 课程归属关系不明确

这个课时属于哪个课程？是独立单课还是系列课的一部分？教学团队以此规划课程体系，学生以此判断学习路径。

建议在页面顶部增加面包屑或课程归属标签（如「所属课程：Vibe Coding 实战系列」）。

### 课时状态与发布流程

教学团队需要管理课时的生命周期：草稿 → 审核 → 已发布 → 下架。当前页面未体现任何状态标识，无法区分「即将上线」和「已完结」的课时。

建议增加课时状态标签，同时为教学后台设计状态流转接口。

### 学生数据反馈缺失

教学团队需要知道这个课时的实际效果：多少人看过、完课率、平均观看时长、课后评价。这些数据不在课时页面上体现，但应在教学后台与课时关联。

建议在技术方案中为课时绑定统计数据接口。

### 内容更新机制

教学团队会迭代课程内容——修正错误、补充案例、更新截图。当前页面是静态内容，未考虑版本备注或更新日志。

建议在课时信息区增加「最后更新于 xxxx 年 xx 月」或 changelog 入口。
